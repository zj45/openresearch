# AGENTS.md - Agent Coding Guidelines

## General

- To regenerate the JavaScript SDK, run `./packages/sdk/js/script/build.ts`.
- ALWAYS USE PARALLEL TOOLS WHEN APPLICABLE.
- The default branch in this repo is `dev`.
- Local `main` ref may not exist; use `dev` or `origin/dev` for diffs.
- Tests cannot run from repo root (guard: `do-not-run-tests-from-root`); run from package dirs.

## Build/Lint/Test Commands

### Root (monorepo)

```bash
bun run typecheck     # Run typecheck across all packages (turbo)
```

### packages/opencode (main CLI)

```bash
bun run test                              # Run all tests (bun test --timeout 30000)
bun test <file>                           # Run single test file
bun test --watch <file>                   # Run single test in watch mode
bun run typecheck                         # TypeScript check
bun run build                             # Build the CLI
bun run db                                # Drizzle DB commands
```

```bash

bun test:unit --watch                     # Watch mode

bun test:e2e                              # Playwright e2e tests

bun test:e2e:local                        # Local e2e testing
bun run typecheck                         # TypeScript check

```

### packages/desktop

```bash
bun run typecheck                         # TypeScript check

bun tauri dev                             # Tauri dev mode
```

- Test actual implementation, do not duplicate logic into tests
- Use `tmpdir` fixture from `packages/opencode/test/fixture/fixture.ts` for tests needing filesystem

## Style Guide

### General Principles

- Keep things in one function unless composable or reusable
- Avoid `try`/`catch` where possible
- Avoid using the `any` type
- Prefer single word variable names where possible
- Use Bun APIs when possible, like `Bun.file()`
- Rely on type inference when possible; avoid explicit type annotations or interfaces unless necessary
- Prefer functional array methods (flatMap, filter, map) over for loops

### Naming (MANDATORY)

- Use single word names by default for new locals, params, and helper functions
- Multi-word names are allowed only when a single word would be unclear or ambiguous
- Do not introduce new camelCase compounds when a short single-word alternative is clear
- Good short names: `pid`, `cfg`, `err`, `opts`, `dir`, `root`, `child`, `state`, `timeout`

```ts
// Good
const foo = 1
function journal(dir: string) {}

// Bad
const fooBar = 1
function prepareJournal(dir: string) {}
```

Reduce total variable count by inlining when a value is only used once.

```ts
// Good
const journal = await Bun.file(path.join(dir, "journal.json")).json()

// Bad
const journalPath = path.join(dir, "journal.json")
const journal = await Bun.file(journalPath).json()
```

### Imports

Group imports in this order (separate with blank lines):
1. Node built-ins (path, fs, etc.)
2. External packages
3. Workspace packages (`@opencode-ai/*`, workspace:*`)
4. Relative imports

### Destructuring

Avoid unnecessary destructuring. Use dot notation to preserve context.

### Variables

Prefer `const` over `let`. Use ternaries or early returns instead of reassignment.

### Control Flow

Avoid `else` statements. Prefer early returns.

### Error Handling

- Use Result types or early returns for error handling
- Avoid try/catch unless absolutely necessary (e.g., parsing untrusted input)
- Propagate errors up rather than swallowing them silently

### Schema Definitions (Drizzle)

Use snake_case for field names so column names don't need to be redefined as strings.

### Database

- **Schema**: Drizzle schema lives in `packages/opencode/src/**/*.sql.ts`
- **Naming**: tables and columns use snake_case; join columns are `<entity>_id`; indexes are `<table>_<column>_idx`
- **Migrations**: generated by Drizzle Kit using `drizzle.config.ts`
- **Tests**: migration tests should read per-folder layout (no `_journal.json`)

## App-Specific Notes

### SolidJS

- Always prefer `createStore` over multiple `createSignal` calls

### Desktop

- Never call `invoke` manually; use generated bindings in `packages/desktop/src/bindings.ts`

### Browser Automation

Use `agent-browser` for web automation:
1. `agent-browser open <url>` - Navigate to page
2. `agent-browser snapshot -i` - Get interactive elements with refs (@e1, @e2)
3. `agent-browser click @e1` / `fill @e2 "text"` - Interact using refs
4. Re-snapshot after page changes
